NID-04
======

Private Replies, Comments and other
-----------------------------------

`draft` `optional`

This NIP defines a method for sending any nostr event privately to a specific recipient by utilizing the gift-wrapping mechanism defined in [NIP-59](59.md) and the relay discovery mechanism defined in [NIP-17](17.md).

## Motivation

Currently, [NIP-59](59.md) gift wrapping is primarily used for direct messages ([NIP-17](17.md), kind `14`/`15`). However, there are many interactions on nostr that users may want to keep private: replies, comments, reactions, and other event kinds. This NIP extends the use of gift wrapping to any event kind, enabling "private nostr" where the protocol works exactly the same, but events are wrapped in a gift wrap before being sent to the recipient's DM relays.

## Overview

A private reply is a regular nostr event (of any supported kind) that is:

1. Left unsigned (turned into a **rumor** as defined in [NIP-59](59.md))
2. Sealed and gift-wrapped (kind `1059`) for each intended recipient
3. Sent to the recipient's DM relays (kind `10050` as defined in [NIP-17](17.md))
4. Also wrapped and sent to the sender's own DM relays for backup/recovery

The inner rumor event is constructed exactly as it would be if it were a normal public event, same kind, same tags, same content. The only difference is that it is never signed and never published to regular relays.

## Supported Event Kinds

Any event kind can be sent as a private interaction. Common use cases include:

- **Kind `1`** - Private replies to posts
- **Kind `1111`** - Private comments ([NIP-22](22.md))
- **Kind `1244`** - Private voice message comments
- **Kind `7`** - Private reactions

The recipient's client unwraps the gift wrap and processes the inner rumor exactly as it would process a regular event of that kind.

## Protocol Flow

### Sending a Private Interaction

1. **Construct the event normally** - Create the event with the appropriate kind, tags (reply tags, mention tags, NIP-22 scope tags, etc.), and content, exactly as if it were a public event.

2. **Create a rumor** - Remove the signature from the event. The event MUST have an `id` computed from its serialized content, but the `sig` field MUST be empty (`""`).

3. **Gift-wrap for each recipient** - Using [NIP-59](59.md):
   - Create a seal (kind `13`) encrypting the rumor with the sender's key to the recipient's pubkey
   - Create a gift wrap (kind `1059`) encrypting the seal with a one-time-use key
   - The gift wrap's `p` tag MUST contain the recipient's pubkey for routing

4. **Gift-wrap for self** - Create an additional gift wrap addressed to the sender's own pubkey. This allows the sender to recover their own private interactions.

5. **Send to DM relays** - Send each gift wrap to the respective recipient's DM relays (discovered via their kind `10050` event as specified in [NIP-17](17.md)). The sender's self-addressed wrap is sent to their own DM relays.

### Receiving a Private Interaction

1. **Fetch gift wraps** - Subscribe to kind `1059` events on own DM relays (this is the same subscription used for NIP-17 DMs).

2. **Unwrap** - Decrypt the gift wrap and seal as specified in [NIP-59](59.md) to obtain the inner rumor.

3. **Validate the rumor** - Verify the rumor has no signature (`sig` is empty), confirming it is a proper rumor. Verify the rumor's pubkey matches the seal's pubkey to prevent impersonation.

4. **Process normally** - Handle the rumor event according to its kind, exactly as a regular event would be handled. For example, a kind `1` rumor with reply tags is displayed as a reply in the thread it references.

## Client Behavior

### Displaying Private Interactions

- Private interactions (rumors received via gift wrap) SHOULD be displayed alongside regular events of the same kind. A kind `1` private reply appears in the same thread as public replies.
- Clients SHOULD visually indicate that an interaction is private (e.g., a lock icon or "private" label).

### Restricting Actions on Private Interactions

- Clients MUST NOT allow reposting (kind `6`) of private interactions, as this would expose the content publicly.
- Clients MUST NOT allow quote-posting of private interactions.
- When replying to a private interaction, clients SHOULD default to sending the reply as a private interaction as well, to maintain the privacy of the conversation.

### Relay Discovery

- Clients MUST use the recipient's kind `10050` relay list to determine where to send gift wraps, consistent with [NIP-17](17.md).
- If the recipient has no kind `10050` event, the private interaction option SHOULD NOT be offered.
- Clients SHOULD also send a self-addressed gift wrap to the sender's own DM relays for recovery purposes.

## Compatibility

- This NIP builds entirely on existing primitives: [NIP-59](59.md) for gift wrapping and [NIP-17](17.md) for relay discovery.
- No new event kinds are introduced. The gift wrap is always kind `1059`, and the inner rumor retains its original kind.
- Clients that already support unwrapping kind `1059` and processing the contained rumor kinds will automatically support private interactions without additional changes.
- The set of rumor kinds a client supports is implementation-defined. At minimum, clients implementing this NIP SHOULD support kind `1` (text notes) as private interactions.
